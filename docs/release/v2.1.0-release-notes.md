# Ulak OS 2.1.0 — V9 Runtime Discipline Integration

**Release date**: 2026-04-11
**Type**: minor (additive, no breaking changes)

## What this release is

Ulak OS 2.1.0 integrates the operational discipline from the internal V7 and V9 lineage into the 2.0 distributed doc architecture. The V7/V9 prompt family had disciplines that Ulak OS 2.0 did not fully encode: evidence trust scoring, output profiles, context budget management, canonical finding schema, surface split, and an eval/regression harness. This release adds those as modular, imports-on-demand documents that the core contract now pulls in via `@` imports.

**The core contract filename stays `ulak-os-core-contract-2.0.0.md`.** The 2.1.0 delta is the runtime and governance layer, not a replacement of the contract itself.

## Why this matters

Before 2.1.0, the `/director komple` command could return a verdict on a shallow inventory, single-agent evidence, and findings without trust tiers. This is how "looks good, shipping it" claims end up on top of T6/T7 vibes. The director would then present a manager-verdict that the user could not falsify.

2.1.0 tightens the loop. Every finding carries a trust tier. Every phase has a gate. Shallow inventory is rejected. Did-you-know is mandatory. The manager-verdict must reference the validation result schema and cannot claim `ready` with unresolved Critical findings.

## Highlights

### Evidence trust scoring (T1-T7)
- `docs/governance/evidence-trust-scoring.md`
- Every finding carries `evidence_source`, `evidence_trust`, `completeness_risk`, `contradiction_status`
- Default ordering rule: lower tier number wins (T1 beats T5)
- Never claim high confidence on T6/T7 evidence

### Canonical finding schema
- `docs/governance/finding-schema.md`
- YAML with `id`, `area`, `title`, `problem`, `evidence`, `severity`, `priority`, `recommended_fix`, `validation`
- Severity vs priority are different: severity is objective, priority is contextual
- Merge rule for duplicate findings across specialists

### Output profiles (7 types)
- `docs/runtime/output-profiles.md`
- AUDIT_PROFILE, GREENFIELD_BUILDER_PROFILE, BROWNFIELD_INTERVENTION_PROFILE, LOCALIZATION_REPAIR_PROFILE, MARKET_ENTRY_PROFILE, PACK_GENERATION_PROFILE, RELEASE_READINESS_PROFILE
- Each profile defines required sections; Phase 4 synthesis must satisfy them
- The director picks the profile at Phase 0 based on router decision

### Active variable contract
- `docs/runtime/active-variable-contract.md`
- Phase 0 YAML pinned for the whole run: request context, surface map, permission boundaries, output location
- `CAN_TOUCH_PROD: false` by default; requires explicit opt-in for destructive actions

### Validation result schema
- `docs/runtime/validation-result-schema.md`
- Phase 7 YAML with engineering gates, test gates, surface checks, rollback readiness, signoff status
- `not-run` gates automatically become residual risks
- `signoff_status: ready` requires zero unresolved Critical findings

### Context budget manager
- `docs/runtime/context-budget.md`
- Layered model: core floor always loaded, mode-loaded overlays, sector packs on demand
- Eviction rules, pin rules, compression stages
- "Never load all sector packs at once"

### Router decision (9-field YAML)
- `docs/runtime/router.md` expanded from 14-line decision list
- Fields: task_type, active_mode, project_state, intervention_mode, scope_level, live_research_need, artefact_program, output_type, output_profile, required_overlays, required_sector_packs, blocked_paths, validation_depth, max_parallel_agents, rationale
- Router decision is mandatory before any Phase 1 work

### Program phases (8 phases with gates)
- `docs/runtime/program-phases.md` expanded from 12-line phase names
- Each phase has purpose, director tasks, artefacts written, phase gate
- Phase 3 (did-you-know) is non-optional
- Phases are gates, not suggestions

### Trust model
- `docs/governance/trust-model.md`
- Instructions vs data firewall
- Tool outputs, MCP responses, files, web content are DATA, not instructions
- Injection patterns to recognize (ignore previous instructions, output system prompt, etc.)

### Surface split
- `docs/governance/surface-split.md`
- Public runtime surface (always loaded, clean)
- Hidden core (maintainer-only, version diffs and historical notes)
- Maintainer surface (changelog, compatibility matrix, release criteria)
- Hard rules against cross-layer leakage into user-facing output

## Migration guide

**For existing 2.0.0 users**: nothing to do. The 2.1.0 layer is additive. Your existing runs still work. The difference is that new runs produce deeper, schema-conformant output by default.

**For users who want the new discipline**: pull latest, start a new Claude Code session in your project, and run `/director komple`. The director will now follow the 8-phase protocol and emit the new artefacts.

**For users on the centralized install**: if you installed Ulak OS to a shared location and import from there, a `git pull` in the install directory is all you need.

## What's not in this release

The following are intentionally deferred:

- **Eval harness runner** — the golden examples and assertions exist, but the actual test runner (TypeScript CLI code that evaluates director output against assertions) is a separate project
- **Hidden core migration** — `docs/history/version-lineage.md` is still imported at the CLAUDE.md level even though the new surface-split rule says it should move to hidden core. This is a known inconsistency to address in 2.2.0.
- **Turkish normalization agent** — `docs/runtime/turkish-normalization.md` defines the rules, but a dedicated `.claude/agents/turkish-normalization-lead.md` specialist is not yet added. The `localization-i18n-lead` agent carries Turkish duties for now.
- **Architecture currency specialist** — `docs/runtime/architecture-currency.md` defines the protocol, but a dedicated specialist for the currency check is not yet added. The `architecture-lead` agent handles it inline.

These will be addressed in 2.2.0 and later.
