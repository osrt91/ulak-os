# SEC-INCIDENT-2026-04-21 — Credentials leaked in public `v2.1.4` git tag

**Severity:** CRITICAL
**Status:** ACKNOWLEDGED · OPERATOR ACTION REQUIRED
**Discovered:** 2026-04-21 (security-redteam final-test pass, pre-v1.0.0-public-GA)
**Reporter:** `security-redteam` subagent
**Affected tag:** `v2.1.4` (commit `4f2f5cf`, file `.gitleaks.toml:34-46`)
**Mitigation at HEAD:** Keys were stripped from the main-branch HEAD in commit `d1d05d6` (v2.1.4 → v2.2.0 window). HEAD is clean.
**Residual exposure:** The public `v2.1.4` tag remains fetchable from `origin` (`github.com/osrt91/ulak-os`). Anyone can run `git show 4f2f5cf:.gitleaks.toml` and extract the full values.

## What leaked

- **Resend API key** — full 35-char key, prefix `re_J...` — outbound email send rights under the owner's Resend account.
- **Cloudflare API token** — 36-char token, prefix `cfk_...` — scope unknown from prefix alone; assume **full account** unless operator confirms otherwise.
- **VPS IPv4** — `72.61.178.25` — public metadata, not a secret, but aids targeting.

## Exposure window

- **Leaked:** within v2.1.4 tag body (Apr 2026)
- **Noticed:** commit `d1d05d6` self-documents the operator noticed and stripped HEAD; the commit message says "rotate Resend immediately"
- **Rotated:** NOT CONFIRMED — no rotation log in `reports/current/secret-rotation-log.md`
- **Still recoverable on origin:** YES (as of 2026-04-21)

Conservative assumption: the keys are **burned**. Treat them as compromised regardless of whether an attacker actually extracted them.

## Required operator actions (before v1.0.0 public GA push)

### 1. Rotate the keys NOW (blocking for v1.0.0 push)

- **Resend**: dashboard → API Keys → revoke the leaked key → issue a new key → update downstream consumers' env files (operator-local, not in this repo)
- **Cloudflare**: dashboard → My Profile → API Tokens → revoke the leaked token → issue a new one with **minimum-necessary scope** (DNS-edit only for ulak-os domain, NOT global) → update downstream consumers

### 2. Decide the git-history remediation path

**Option A — Accept the keys as burned (DEFAULT, lowest risk)**

- Do nothing to git history
- Verify post-rotation the old keys are truly dead (revoked → confirm 401 on sample request)
- Document in this file that rotation happened and when

**Option B — Rewrite history (destructive, coordination cost)**

- Run `git filter-repo --replace-text patterns.txt` with the two keys in the patterns file
- Force-push `main` + all tags to `origin`
- Every existing fork/clone breaks (must re-clone); coordination required
- Only choose this if an active attacker scenario is detected or if leaked scope can't be fully revoked

**Recommendation: Option A.** History rewrite is usually worse than acceptance. The keys are dead once rotated and revoked; no further action is needed on the tag itself.

### 3. Confirm rotation in writing

Add a dated entry to `reports/current/secret-rotation-log.md`:

```

2026-04-21 — Cloudflare API token rotated (old: cfk_...); operator-verified revocation; new token scope: DNS-edit only.
```

Without this file, `release-readiness-auditor` should continue to flag the incident.

### 4. Verify once more with a deep scan

(If this script doesn't exist yet, `gitleaks detect --no-git -v` against the repo HEAD and `gitleaks git` against `--all` refs is equivalent.)

## Governance lessons

This incident maps to anti-patterns:

- **AP-16** — `.env.local` git-tracked (the original v2.1.4 leak was an `.env.local`-equivalent)
- **AP-19** — secrets in files at root-level or in gitleaks config allowlists

The `docs/governance/secrets-rotation-policy.md` document correctly identified this class of failure; the gap was in **pre-commit enforcement**, not in the policy text.
